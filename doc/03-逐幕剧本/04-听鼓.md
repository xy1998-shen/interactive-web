# 04 听鼓

## 场景描述

上一幕结尾，窗外出现淡朱砂鼓波。用户点击“入下一幕”后，鼓波沿水纹扩散到楚江水面，画面从临水屋舍回到略深的雾青江面。远山云气和暖金余光仍然可见，江面不进入全黑夜景。

画面采用人在船尾的鼓手第一视角：主背景一体呈现龙舟内部、前方鼓手、合桨人群、江面和两岸山雾。用户像站在船尾随鼓行舟；鼓声、江水溅起、水痕浮字和知识卡片叠加在这一张完整背景上，不再额外叠加第二套船头或鼓面前景，避免两个视角互相冲突。完成后，龙舟驶过留下长水痕，水痕中浮出字粒，引出下一幕“问诗”。

## 目的

- 形成全篇从静态手作转入江上竞渡的动感高潮。
- 用节奏表现龙舟竞渡和众人合力，但避免做成竞技小游戏。
- 承接第三幕的远处鼓波，不再强行做“粽形变船头”的硬形变。
- 让用户从鼓手第一视角感到“敲一下，江面向前走一步”。
- 用水痕字粒自然引出第五幕“问诗”。

## 交互

1. 入场时，画面显示短句“远鼓入江，舟影渐明。”，总时长控制在 600ms 至 900ms。
2. 系统自动播放 6 下由慢到快的鼓声序列，推动背景、江水溅起、水痕浮字和船行惯性逐步增强。
3. 用户点击画面只触发较轻的水面回声和船身惯性反馈，不额外增加鼓声次数，也不作为完成条件。
4. 每次鼓声只短暂浮现 1 个动作字，如“起”“合”“渡”“破”“和”“止”；字必须从左右两侧江面浮出，不能出现在船身或人物身上。
5. 字出现时，下方同步出现江水溅起和椭圆水纹。水花、短浪、水纹和字粒必须共用同一组江面坐标，避免水花在舟上、字在水里的错位。
6. 随自动节奏推进，提示依次变化为“远鼓自江上”“舟随鼓行”“众桨同频”“鼓声已渡”。
7. 自动序列结束后，鼓声把舟影送过江面，左右江面浮出“江”“艾”“舟”“鼓”“风”等字粒，作为第五幕“问诗”的入口预告。
8. 完成后自动出现和“裹青”同款的右下知识卡片，介绍“听鼓竞渡 · 舟鼓同频”；不再使用水痕光点点击展开知识。
9. 用户点击「入下一幕 →」后，水痕扩散为深青静水，进入“问诗”。

播放过程中不显示分数、倒计时、排名、失败或“连击”文字。完成条件由自动鼓序列控制，用户看到的是鼓声推进、水路向前和水痕变长，而不是数字进度。

## 技术

- 点击区域覆盖背景里的鼓手 / 船行视觉中心，保证可用性。
- 鼓点反馈使用 Pixi Graphics 生成江水溅起：短水线、水滴和椭圆白浪组成，600ms 左右扩散并淡出。不再使用醒目的朱砂圆环作为主反馈。
- 主背景和水痕使用缓慢行进、轻微缩放、纵摇和鼓点惯性组合，不做整张图硬平移回弹；自动鼓序列结束后完成。
- 背景按 8% 过扫铺设，给纵摇和 resize 留出边界，避免露出底色条。
- 不叠加 `已删除的听鼓旧方案备用素材` 或 `已删除的听鼓旧方案备用素材` 这类独立船头前景；当前主背景已经包含船尾第一视角、鼓手和合桨人群。
- 用户加击不参与节奏判定和完成进度，只补充即时声画反馈。
- 水痕、鼓点字和完成字粒优先用 Pixi Graphics / Text 代码生成。字粒统一落在左右江面水痕上，下面带轻水纹，不能沿船身中线排布。
- 鼓面点击时使用 Web Audio 合成约 120ms 的低频包络音；龙舟过江时可叠加一次更开阔的低频衰减。
- 轻知识沿用“裹青”同款 DOM 知识卡片：完成后自动出现，卡片展示后隐藏底部完成提示，避免同屏文字过多。
- 场景完成条件：龙舟过江、水痕和字粒出现、推进入口显示完成。

## 实现经验

本幕最终采用“完整背景 + 代码反馈”的实现，不再维护旧版“背景 + 船头鼓前景 + 鼓槌”的拆层方案。后续维护时优先遵守以下口径：

- **素材只认一张主背景**：运行时第四幕资源只加载 `drumBgFirstperson`，对应 `assets/backgrounds/drum-bg-firstperson.png`。`已删除的听鼓旧方案备用素材`、`已删除的听鼓旧方案备用素材`、`已删除的听鼓旧方案备用素材` 和 `已删除的听鼓旧方案备用素材` 只能作为历史备用素材记录，不得重新加入第四幕 preload 或渲染层。
- **不要混用两个视角**：`drum-bg-firstperson.png` 已经包含人在船尾、前方鼓手、合桨人群和江面纵深。如果再叠加船头鼓前景，会形成“背景里一个鼓手视角 + 前景另一个船头鼓视角”的空间冲突。
- **运动不要硬平移**：整张背景不能随鼓点大幅横跳再回弹。船行反馈应由轻微缩放前冲、纵向惯性、低幅横摆和水痕层独立滚动共同完成，让画面像在水面上受力，而不是像图片被机械拖动。
- **背景必须过扫**：16:9 背景在 16:9 视口下几乎没有垂直余量。第四幕需要在 `fitCover` 后额外 x/y 过扫，给纵摇、鼓点惯性和窗口 resize 留安全边界，避免露出纯色底。
- **resize 后重算基准**：第四幕会在每帧根据 `bgBaseScaleX/Y`、`bgCenterX/Y` 计算背景位置。窗口尺寸变化后必须重新执行第四幕专用 layout，并重新捕获基准值；否则 base-scene 的通用 `fitCover` 会被下一帧旧基准覆盖。
- **自动节奏和用户加击分开**：完成条件由自动鼓序列控制。用户点击只触发额外鼓波、音频和惯性反馈，不递增完成进度，也不参与稳定节奏判定，避免把这一幕重新做成点击计数游戏。
- **临时 Pixi 对象要自销毁**：鼓波、水花、字粒等循环生成对象的 `onComplete` 必须销毁当前局部对象。不要把循环变量暴露给异步回调，避免快速鼓点下销毁错对象或漏对象。
- **空值与恢复态要防守**：`beatWords`、`wakeWords`、`knowledgeText`、`knowledgeDot` 等都要允许为空或已销毁。用户可能从完成态恢复、快速切换场景或走资源降级路径，相关函数不能假设对象必然存在。
- **缓存验证要注意脚本 URL**：如果本地预览仍提示旧资源 key，例如 `drumFgBoatFrame`，优先检查 HTML 中脚本是否被浏览器缓存。只给页面 URL 加 query 不一定刷新脚本文件；必要时给相关 `script src` 加版本参数。
- **鼓声次数只认配置源**：鼓声总数以 `CONFIG.drum.totalBeats` 为唯一配置源。自动序列、提示节点和完成判定都从这个值派生，避免出现“配置 6 下、实际播放更多下”的漂移。
- **字和水花共用江面坐标**：鼓点字、水花、短浪和椭圆水纹必须先从同一个江面坐标函数取点，再分别渲染。不要让水花继续使用画面中心或鼓手中心坐标，否则会出现“字在水里、水花在舟上”的错位。
- **字粒必须避开船身和卡片**：过程字与完成字都落在左右两侧江面水痕上，不在船身中线横排。右侧“鼓”等字需高于右下知识卡片区域，避免被卡片遮挡。
- **可读性靠水墨层级解决**：字粒用亮米白填充、深青描边和轻水纹底座增强可读性，不使用红色描边、大红圈或调试标注式边框。
- **知识展示使用 DOM 卡片**：第四幕完成后自动展示和“裹青”同款的右下知识卡片，同时隐藏底部完成提示，只保留「入下一幕 →」入口。

## 文案

- 入场短句：远鼓入江，舟影渐明。
- 动作提示：随鼓而行 / 舟随鼓行 / 众桨同频。
- 鼓点动作字：起、合、渡、破、和、止。
- 过程轻知识：鼓为令，众桨同频。
- 过程轻知识：竞渡从江上追思，成为五月节庆。
- 核心诗性句：一声鼓起，百桨入江。
- 完成短句：鼓声已渡。
- 知识卡片标题：听鼓竞渡 · 舟鼓同频。
- 知识卡片正文：竞渡时鼓点统一节奏，鼓手居前发令，桨手随声入水，整舟才能同频向前。龙舟竞渡承接江上追思，也把乡人合力、祈愿安康的节令情感，化成可听见的鼓声与水痕。

### 诗句呈现

第四幕节奏最快，不放长画轴，也不做大段字幕。文字应跟着江水溅起和水痕出现：

1. 每次鼓声只出现 1 个动作字，像鼓声击开江面。
2. 字粒必须落在左右江水水痕上，并与下方水花、水纹共享同一个江面锚点。
3. 龙舟过江时，“江”“艾”“舟”“鼓”“风”作为水痕字粒分布在两侧江面，不在船身上横排。
4. 完成态不再保留大段浮字；文化解释交给右下知识卡片，避免和下一幕问诗抢焦点。
5. 水痕字粒只做第五幕预告，不在本幕组成完整诗句。

## 验收要点

- 入场能明确承接第三幕的淡朱砂鼓波，而不是突兀切到独立竞渡画面。
- 画面仍属于雾青江光体系，不突然全黑，不变成高饱和红色节庆海报。
- 自动鼓声反馈必须即时：江水溅起、水花、低频鼓声和第一视角推进同步出现。
- 背景运动应像船行水面上的惯性纵摇和缓慢前进，不能出现规律性横跳或整图机械回弹。
- 朱砂红只保留为背景和船身中的极少量节奏点缀，主反馈改用雾白、浅水青和米白水花，不能出现突兀红色圈圈。
- 第一视角推进有起鼓、合桨、过江三个层次，但不出现游戏化 UI。
- 轻知识必须使用右下知识卡片呈现，文案聚焦“以鼓为令、众桨同频”，不使用水痕光点。
- 完成态先显示水痕和字粒，再出现「入下一幕 →」入口，不自动硬切。
- 完成后水痕字粒必须明确落在江面上，且不能被右下知识卡片遮挡。
- 本地静态服务打开时背景、鼓面、龙舟、鼓波、声音降级和字粒都正常可用。
