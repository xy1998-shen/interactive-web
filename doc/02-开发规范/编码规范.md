# 编码规范

## 注释

- 关键逻辑需要写注释，帮助理解代码意图。
- 注释保持简短，只说明必要背景、原因或注意事项。
- 避免写重复代码含义的注释，例如逐行解释变量赋值。

## 文件组织

- 保证单一文件不过长，避免把常量、配置、工具函数和业务逻辑全部堆在同一个文件中。
- 当文件内容开始变得难以快速浏览时，应按职责拆分。
- 拆分后保持命名清晰，让文件名能直接表达用途。
- 单个 JavaScript 文件建议控制在 300 行以内，超过 500 行应优先拆分。
- 单个 CSS 文件建议控制在 300 行以内，超过后按基础样式、布局、组件、动画或页面模块拆分。
- 每个函数只做一件明确的事，函数过长或同时处理多种职责时应拆成小函数。

## 可拆分内容

- 常量可以抽到 `constants.js`。
- 配置可以抽到 `config.js`。
- 通用功能函数可以抽到 `utils.js`。
- 只被单个模块使用的内容，可以优先放在该模块附近，避免过度全局化。
- 资源清单、场景配置、动画参数等可配置内容不要散落在业务逻辑中。
- 复杂模块可以按职责拆成 `loader.js`、`scene-manager.js`、`dom-layer.js`、`scenes/` 等文件。

## 命名

- 变量、函数、文件名使用能表达业务含义的名称，避免 `data`、`temp`、`handle` 这类含糊命名。
- 常量使用全大写或清晰的常量名，例如 `SCENES`、`ASSET_MANIFEST`。
- 布尔值命名体现状态含义，例如 `isLoaded`、`hasError`、`canPlay`。
- 事件处理函数使用动作命名，例如 `onResize`、`handleSceneClick`。

## 状态与副作用

- 状态集中管理，避免多个模块随意读写同一份状态。
- 修改状态后要同步更新对应 UI，避免状态和界面表现不一致。
- DOM 查询、事件绑定、资源加载、动画启动等副作用应放在明确的初始化或生命周期函数中。
- 进入和退出场景时要清理事件、ticker、定时器和临时对象，避免重复绑定或内存泄漏。

## 错误处理

- 异步加载资源必须处理失败情况，并给出可定位的问题信息。
- 不吞掉错误；至少要记录必要上下文，例如资源 key、场景 id、当前步骤。
- 用户可感知的失败要有兜底表现，避免页面空白或交互无响应。

## 资源与性能

- 图片、纹理、音频等资源统一走资源清单管理，避免在代码中到处写路径。
- 大资源按需加载，下一场景资源可以预加载，但不要一次性加载所有无关资源。
- 高频更新逻辑中避免反复创建对象、查询 DOM 或触发布局计算。
- 动画参数集中配置，避免魔法数字散落在多个函数里。

## CSS 规范

- 颜色、字号、间距、阴影等设计变量优先放在 `:root` 中统一管理。
- 选择器保持简洁，避免过深嵌套和过强优先级。
- 组件样式按模块集中书写，状态类统一使用 `is-` 或 `has-` 前缀。
- 响应式规则要和组件放在相邻位置，方便维护同一模块的不同屏幕表现。

## 兼容与可访问性

- 可点击元素优先使用语义化按钮，并保留键盘焦点样式。
- 关键交互需要同时考虑鼠标、触摸和键盘场景。
- 文案、按钮和进度状态要避免只依赖颜色表达。
- 移动端和窄屏下要确认文字不溢出、不遮挡核心交互区域。

## 提交前检查

- 打开页面确认无控制台错误。
- 检查核心流程至少能完整走通一次。
- 检查新增或修改资源路径是否真实存在。
- 检查长文件、重复逻辑、魔法数字是否有必要继续拆分。
- 删除临时调试代码、无用注释和未使用变量。

## 基本原则

- 每个文件只承担清晰、单一的职责。
- 公共逻辑复用前先确认确实有复用价值。
- 修改代码时同步维护相关注释、常量、配置和工具函数，避免文档或命名失真。
- 优先做小而明确的改动，避免为了局部需求引入大范围重构。
- 代码要能让下一位维护者快速判断入口、状态、资源和渲染流程。
