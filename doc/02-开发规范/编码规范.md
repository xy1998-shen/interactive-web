# 编码规范

## 注释

- 关键逻辑需要写注释，帮助理解代码意图。
- 注释保持简短，只说明必要背景、原因或注意事项。
- 避免写重复代码含义的注释，例如逐行解释变量赋值。

## 文件组织

- 保证单一文件不过长，避免把常量、配置、工具函数和业务逻辑全部堆在同一个文件中。
- 当文件内容开始变得难以快速浏览时，应按职责拆分。
- 拆分后保持命名清晰，让文件名能直接表达用途。
- 单个 JavaScript 文件建议控制在 300 行以内，超过 500 行应优先拆分。
- 单个 CSS 文件建议控制在 300 行以内，超过后按基础样式、布局、组件、动画或页面模块拆分。
- 每个函数只做一件明确的事，函数过长或同时处理多种职责时应拆成小函数。

## 可拆分内容

- 常量可以抽到 `constants.js`。
- 配置可以抽到 `config.js`。
- 通用功能函数可以抽到 `utils.js`。
- 只被单个模块使用的内容，可以优先放在该模块附近，避免过度全局化。
- 资源清单、场景配置、动画参数等可配置内容不要散落在业务逻辑中。
- 复杂模块可以按职责拆成 `loader.js`、`scene-manager.js`、`dom-layer.js`、`scenes/` 等文件。

## 命名

- 变量、函数、文件名使用能表达业务含义的名称，避免 `data`、`temp`、`handle` 这类含糊命名。
- 常量使用全大写或清晰的常量名，例如 `SCENES`、`ASSET_MANIFEST`。
- 布尔值命名体现状态含义，例如 `isLoaded`、`hasError`、`canPlay`。
- 事件处理函数使用动作命名，例如 `onResize`、`handleSceneClick`。

## 状态与副作用

- 状态集中管理，避免多个模块随意读写同一份状态。
- 修改状态后要同步更新对应 UI，避免状态和界面表现不一致。
- DOM 查询、事件绑定、资源加载、动画启动等副作用应放在明确的初始化或生命周期函数中。
- 进入和退出场景时要清理事件、ticker、定时器和临时对象，避免重复绑定或内存泄漏。

## Pixi / GSAP 动画状态

- 对同一个 Pixi 对象做连续动画时，进入场景时先记录基准值，例如 `baseX`、`baseY`、`baseScaleX`、`baseScaleY`。后续 `fromTo` 和完成态动画都应基于基准值计算，避免快速交互导致位移或缩放漂移。
- 同一个属性被新动画接管前，应先 `killTweensOf` 对应对象或子对象，例如 `sprite`、`sprite.scale`、`container`。尤其是点击反馈和完成退场同时操作 `x/y/alpha/scale` 时，完成态动画必须先接管旧 tween。
- Pixi 的缩放应优先补间 `sprite.scale.x/y`，不要把 DOM 风格的 `scaleX/scaleY` 当作 Pixi 属性使用。
- 场景中的延迟回调统一走公共封装，例如 `scheduleCall(delay, fn)`。裸 `gsap.delayedCall` 必须保存句柄并放入 `cleanups`，否则切场景后旧回调可能继续 `finish()`、`goTo()` 或操作已销毁的 Pixi 对象。
- 循环里创建的临时 Pixi 对象，如果在 `onComplete` 中销毁，应确保闭包引用的是当前对象。可以用 `forEach`、局部函数或 IIFE，避免所有回调引用到同一个循环变量。
- 动画配置数组允许为空时，入口函数要先做防御检查。例如字粒、提示文案、素材列表为空时直接返回，不让 `% length`、索引访问或 `undefined` 文案进入渲染逻辑。
- 完成态、恢复态和降级态都要做空值防护。公共 DOM / Pixi 文本对象可能因场景恢复、跳转或降级路径不存在，调用前先判断对象存在。

## 错误处理

- 异步加载资源必须处理失败情况，并给出可定位的问题信息。
- 不吞掉错误；至少要记录必要上下文，例如资源 key、场景 id、当前步骤。
- 用户可感知的失败要有兜底表现，避免页面空白或交互无响应。

## 资源与性能

- 图片、纹理、音频等资源统一走资源清单管理，避免在代码中到处写路径。
- 大资源按需加载，下一场景资源可以预加载，但不要一次性加载所有无关资源。
- 高频更新逻辑中避免反复创建对象、查询 DOM 或触发布局计算。静态背景的 `fitCover()`、overlay 重绘等只应在视口尺寸变化时执行。
- Web Audio 上下文必须有明确生命周期。场景内创建的 `AudioContext` 应在 `onExit()` 中关闭；如果多场景共用音频能力，必须由公共基类统一创建、恢复和关闭。
- 动画参数集中配置，避免魔法数字散落在多个函数里。

## CSS 规范

- 颜色、字号、间距、阴影等设计变量优先放在 `:root` 中统一管理。
- 选择器保持简洁，避免过深嵌套和过强优先级。
- 组件样式按模块集中书写，状态类统一使用 `is-` 或 `has-` 前缀。
- 响应式规则要和组件放在相邻位置，方便维护同一模块的不同屏幕表现。

## 兼容与可访问性

- 可点击元素优先使用语义化按钮，并保留键盘焦点样式。
- 当前提交版本以 PC 横屏为验收范围，关键交互优先保证鼠标和键盘可用。移动端暂不支持时，不要在验收项中写移动窄屏必须通过。
- 文案、按钮和进度状态要避免只依赖颜色表达。
- 后续若启动移动端适配，应单独补布局、触摸命中区和无 hover 交互方案，不和 PC 提交修复混在同一轮。

## 提交前检查

- 打开页面确认无控制台错误。
- 检查核心流程至少能完整走通一次。
- 检查新增或修改资源路径是否真实存在。
- 检查提交入口文档是否面向评委说明作品、打开方式、推荐路线、验收范围和不支持范围。
- 检查长文件、重复逻辑、魔法数字是否有必要继续拆分。
- 删除临时调试代码、无用注释和未使用变量。

## 基本原则

- 每个文件只承担清晰、单一的职责。
- 公共逻辑复用前先确认确实有复用价值。
- 修改代码时同步维护相关注释、常量、配置和工具函数，避免文档或命名失真。
- 优先做小而明确的改动，避免为了局部需求引入大范围重构。
- 代码要能让下一位维护者快速判断入口、状态、资源和渲染流程。
