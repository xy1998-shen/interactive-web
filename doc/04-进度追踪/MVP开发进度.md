# 《楚江寻艾》MVP 开发进度

## 一、MVP 定义

首个 MVP 的目标不是做完所有高级动效，而是先得到一个**完整、可玩、风格不跑偏、文化信息能闭环**的 PC 端版本。

MVP 范围采用：

**7 幕骨架完整 + 5 幕重点可玩 + 2 幕轻交互收束。**

用户应能从入江一路走到尾声，完成每幕核心动作，并在尾声带走“艾、粽、舟、钟”四个明确记忆点。

## 二、MVP 必须包含

### 1. 完整流程

必须贯通以下 7 幕：

1. 入江
2. 寻艾
3. 裹青
4. 听鼓
5. 问诗
6. 和鸣
7. 尾声 / 端午印记

每一幕必须满足：

- 能进入。
- 有主视觉和章节题签。
- 有一个核心交互或轻交互。
- 能判定完成。
- 能进入下一幕。

### 2. 核心交互

| 分幕 | MVP 交互 | 完成条件 |
| --- | --- | --- |
| 入江 | 点击江面，画轴叙事、斜向雾散、小舟沿左下至右上方向行进 | 小舟到位后画轴保留，点击画面或按任意键进入寻艾 |
| 寻艾 | 拖动艾草到窗前或门楣，触发人物挂艾动画 | 窗前挂艾动画完成并显示画轴故事 |
| 裹青 | 三步折叠简化版 | 左叶、右叶、青线依次完成 |
| 听鼓 | 点击鼓面推动龙舟 | 点击达到指定次数，龙舟驶过 |
| 问诗 | 点击 3 个字粒生成短诗 | 第 3 个字粒被选中并显示结果 |
| 和鸣 | 点击编钟触发声波 | 声波扩散完成 |
| 尾声 | 点亮艾、粽、舟、钟节点 | 四个节点可逐一点亮 |

### 3. 视觉基础

MVP 必须使用 `../03-设计素材/资产映射与开发准则.md` 中定义的资产映射。

最低要求：

- 每幕使用对应背景和核心精灵。
- 标题、短句、题签、知识点全部用 DOM 或 Pixi Text 实现。
- 江流进度线可显示当前进度。
- 首屏保持 `design/第一轮/01.png` 的明亮雾青基调。
- 问诗和尾声不能出现普通信息卡片。

### 4. 轻知识机制

MVP 必须保留轻知识，因为它解决“信息量不够”的问题。

实现范围：

- 寻艾：用户拖动艾草到窗前或门楣，触发人物挂艾动画，画轴讲述采艾、负艾、悬艾守户。
- 裹青：青线光点触发粽子轻知识。
- 听鼓：水痕光点触发龙舟轻知识。
- 问诗：关键词 `屈原 · 端午 · 楚江` 触发诗意注脚。
- 和鸣：声波光点触发编钟礼乐轻知识。
- 尾声：艾、粽、舟、钟四点点亮汇总。

轻知识不阻塞主流程。用户不点击轻知识，也应能继续进入下一幕。

## 三、MVP 暂缓内容

以下内容不进入首个 MVP，避免范围膨胀：

| 暂缓项 | MVP 替代方案 |
| --- | --- |
| 复杂物象变形过渡 | 使用渐隐渐显 + 方向性光纹 |
| 水纹变艾草、粽形变龙舟等高级 morph | 只保留 SVG / 粒子暗示 |
| 高密度粒子系统 | 使用少量水纹、光点、拖尾 |
| 完整声音系统 | 可先静音，最多保留鼓点和编钟基础音 |
| 移动端适配 | 暂只验收 PC 横屏 |
| 字体子集裁切 | 先用系统宋体 / 仿宋气质字体 |
| 复杂性能优化 | 先保证无明显卡顿，再做细化 |
| 最终打包压缩 | MVP 后进入提交前打磨阶段 |

## 五、MVP 验收标准

MVP 通过必须同时满足：

1. **完整性**：7 幕都能进入并完成。
2. **可玩性**：每幕至少有一个可感知交互。
3. **风格一致**：首屏和后续场景保持雾青江光、水墨克制，不回到全黑夜景。
4. **信息闭环**：用户能通过轻知识和尾声点亮理解艾、粽、舟、钟。
5. **本地可运行**：不访问外链，不依赖网络资源。
6. **无明显技术错误**：无资源缺失、无主流程阻塞、无严重布局错位。
7. **PC 优先**：当前只验收 PC 横屏，不要求移动端。

## 六、MVP 不通过条件

出现以下任一情况，不能认为 MVP 完成：

- 只能展示静态页面，无法从头走到尾。
- 某一幕缺少完成条件，流程卡住。
- 使用设计稿整图作为最终页面背景。
- 轻知识全部被删掉，导致文化信息闭环缺失。
- 问诗使用随机句子或在线生成，结果不可控。
- 尾声没有艾、粽、舟、钟四点汇总。
- 页面出现外链、二维码、社交跳转。
- 断网后资源加载失败。

## 七、MVP 后优化池

MVP 通过后，再从下列内容中挑选优先级：

1. 物象变形过渡：水纹到艾草、粽形到船形、字散到声波。
2. 声音系统：水声、艾草摩擦、鼓点、编钟。
3. 粒子打磨：香气轨迹、鼓波、水痕、字粒漂浮。
4. 字体子集：裁切中文字体，控制体积。
5. 性能优化：加载分组、纹理压缩、内存释放。
6. 提交打包：zip、README、离线验证。

## 八、进度更新规则

后续开发中，每完成一个任务，就更新本文对应状态。

更新原则：

- 不把“写了代码”标为完成，必须经过浏览器或离线打开验证。
- 如果采用降级方案，状态写 `降级完成`，并在备注中说明原因。
- 如果发现素材不合适，先更新 `../03-设计素材/资产映射与开发准则.md`，再继续开发。
- MVP 范围不能随意扩大；新增想法先放入“优化池”。
