# 首页拨雾交互改进方案

## 背景

当前首页“入江”场景希望通过“拨开江雾”建立进入《楚江寻艾》的仪式感。页面已有雾层、风痕、标题渐显、江流进度点等元素，视觉氛围成立，但主交互的理解成本偏高。

从第三方用户视角看，当前体验更像“在动态雾气里摸索隐藏规则”，而不是“完成一次清楚的入江动作”。首页作为作品入口，应该优先让用户明确知道：

- 我现在能做什么。
- 我做的动作是否有效。
- 还差多少完成。
- 完成后怎么进入下一步。

## 当前问题

### 1. Hover 风痕和点击拨雾重叠

当前鼠标移动时会产生风吹雾散的短暂效果，按下拖动时也会产生拨雾效果。两者都作用在同一片雾层上，视觉语言相似，用户很难区分：

- 鼠标划过是在预览氛围，还是在推进任务。
- 点击/拖动是在完成主动作，还是只是制造更强的视觉效果。
- 哪一种动作会真正进入下一幕。

结果是用户会把 hover 反馈误认为主交互的一部分，进而觉得触发规则随机。

### 2. 完成规则不可预测

当前进入下一步依赖内部进度阈值，用户看不到明确的完成条件。实际体验中，用户不知道：

- 要拖动多长距离。
- 是否需要连续拖动。
- 是否需要沿某个方向拖动。
- 是否需要拨开指定区域。
- 点击一下有没有意义。

“拨开江雾”作为文案是有诗意的，但作为交互指令不够具体。入口交互不应该依赖用户猜测。

### 3. 已拨开的雾会重新盖上

当前拨雾点有生命周期衰减，雾层会重新覆盖已拨开的区域。这个效果可以制造自然流动感，但会削弱用户的操作反馈：

- 用户刚刚完成的动作被系统撤回。
- 用户无法积累可见成果。
- 进度在内部增加，但视觉上像没有推进。
- 用户会误判为“刚才没有操作成功”。

首页入口应当强调确定性和正反馈。雾可以流动，但已经完成的拨开结果不能完全回退。

### 4. 下一步入口不够明确

完成后目前提示“江雾已开，点击进度线可切换场景”。这个提示有两个问题：

- “进度线”是系统控件语言，不是作品叙事语言。
- 完成后仍要求用户理解底部节点可以点击，路径不够自然。

用户完成拨雾后，应该直接看到一个明确、语义化的下一步入口，例如“入江寻艾”或“继续寻艾”。

## 原因分析

### 1. 氛围交互和任务交互没有分层

当前设计中，hover 风痕和拖动拨雾都是“擦开雾层”。虽然代码上区分了 `windPoints` 和 `fogCuts`，但用户只看到相似的雾气变化，因此无法建立清晰心智模型。

首页的主交互应该只有一个。其他反馈可以存在，但必须明显弱于主交互，并且不能改变主任务状态。

### 2. 系统状态没有外显

内部有 `progress` 和 `completeProgress`，但页面没有把“进度正在增加”和“完成条件接近达成”清楚表达出来。用户看到的是一团会变化的雾，而不是一条可理解的任务路径。

### 3. 视觉诗意优先于操作确定性

“入口交互点隐藏在水纹中”符合意境，但对于第一屏来说风险较高。用户进入作品时还没有学习任何规则，过度隐藏会让第一步变成猜谜。

### 4. 临时效果和永久效果混在一层

短暂风痕、拖动拨雾、薄雾面纱都在同一画面区域里共同变化。缺少“临时预览层”和“永久完成层”的差异，导致拨开的结果看起来会被反复覆盖。

## 改进目标

首页交互改造应满足以下目标：

1. **单一主动作**：用户只需要理解一种核心操作。
2. **触发规则明确**：不用猜测怎样才算成功。
3. **反馈可累计**：用户操作后能看到稳定推进。
4. **完成路径自然**：完成后直接进入或给出明确 CTA。
5. **保留诗意氛围**：不把首页改成普通功能按钮页。

## 方案总览

建议将首页改为：

> **按住并拖动光痕，拨开一片固定江雾；进度达到阈值后雾气稳定散开，出现“入江寻艾”入口。**

核心原则：

- Hover 只做轻微提示，不再制造大面积雾气变化。
- 拖动才推进拨雾进度。
- 已拨开的雾保留，不完全回盖。
- 页面给出方向、进度和完成状态。
- 完成后出现明确的下一步入口，而不是让用户猜进度点。

## 详细方案

### 1. 交互模型

#### 初始态

页面保留当前江面、远山、雾层和标题氛围。

建议把提示文案从单纯的“拨开江雾”调整为更具操作性的短句：

> 按住光痕，拖动拨开江雾

提示位置建议放在画面下方中间，靠近用户自然操作区域。文案不需要做成强按钮，但需要比现在更像可操作提示。

#### Hover 态

鼠标移动到可操作区域时：

- 光痕或水纹轻微呼吸。
- 鼠标附近出现非常弱的高光。
- 不大幅擦开雾层。
- 不增加进度。
- 不触发下一步。

Hover 的作用只回答一个问题：这里可以操作。

#### 拖动态

用户按住并拖动时：

- 光痕跟随指针。
- 雾层沿拖动路径稳定散开。
- 底部或光痕旁出现轻量进度反馈。
- 拖动距离越长，雾散得越明显。

推荐完成规则：

- 累计有效拖动距离达到阈值，完成拨雾。
- 或进度达到 70%，自动进入完成态。
- 不要求用户画出指定形状。
- 不要求精确拖到某个小目标。

这样用户只需要理解“持续拖动即可”。

#### 完成态

完成后：

- 主雾层稳定打开，不再重新覆盖核心视野。
- 标题完全清晰。
- 江面出现下一幕方向的水纹或艾草剪影。
- 出现明确 CTA：`入江寻艾` / `继续寻艾`。

点击 CTA 后进入“寻艾”。

不建议完成后自动跳转。首页完成态需要给用户短暂停留和确认的机会，进入下一幕应由 CTA 点击触发。

### 2. 视觉层级调整

建议把雾气拆成三类：

| 层级 | 作用 | 行为 |
| --- | --- | --- |
| 背景薄雾 | 保持江面氛围 | 缓慢漂移，不参与交互 |
| 主雾幕 | 首页任务对象 | 拖动后稳定散开 |
| Hover 风痕 | 提示可操作 | 很弱、短暂、不计入进度 |

这样用户能感知到：

- 漂移的雾是环境。
- 被拨开的雾是我的成果。
- 鼠标划过的光只是提示。

### 3. 进度反馈

当前底部已有 7 个章节节点，不建议把它继续承担“拨雾进度”职责，因为章节进度和当前动作进度不是一回事。

建议新增一个轻量的当前动作进度反馈：

- 形式一：提示文案下方一条细水纹进度线。
- 形式二：光痕尾部逐渐变亮。
- 形式三：标题附近雾青光晕逐步增强。

不建议使用明显的百分比数字，会破坏诗意。

推荐文案状态：

| 状态 | 文案 |
| --- | --- |
| 初始 | 按住光痕，拖动拨开江雾 |
| 拖动中 | 江雾渐开 |
| 接近完成 | 江路将现 |
| 完成 | 江雾已开，入江寻艾 |

### 4. 下一步入口

完成后不要只提示“点击进度线”。建议在画面中出现一个叙事化入口：

> 入江寻艾 →

入口可以表现为江面光点、水纹题签或艾草剪影，不需要做成普通按钮，但必须具备明显可点击状态：

- hover 有轻微发光。
- cursor 为 pointer。
- focus 可见，支持键盘访问。
- 点击后进入下一幕。

底部章节进度点仍可保留，但它应该是辅助导航，不是主线入口。

### 5. 状态保留规则

拨雾结果需要从“短生命周期点”调整为“永久或半永久路径”：

- 拖动产生的主拨雾路径不再快速衰减。
- 可以保留很慢的边缘回雾，但核心开口不能消失。
- 完成后固定为打开态。
- 如果用户中途松手，已完成的部分仍保留，允许继续拖动补足。

这能让用户明确知道每次操作都有效。

## 实现建议

### 第一阶段：降低混淆

目标是用最小改动先解决”看不懂”的问题。

1. 降低 hover 风痕强度和半径。
   - 当前 wind 的 `centerAlpha` 为 `0.22`，能形成可见擦除洞。建议降至 `0.08~0.10`，让 hover 变成”微微亮”而不是”擦开了一块”。
2. hover 不再形成明显可见的拨雾洞。
3. 拖动拨雾点不快速衰减，或衰减速度大幅降低。
   - 当前 `fogCutLifeDecay: 0.00018`，建议改为 `0.00002` 或直接设为 `0`。
   - 注意：life 不衰减时点不会被视觉淘汰，但 `maxFogCuts: 90` 的上限仍会淘汰最旧的点。如果完全去除衰减，需确认 90 点上限足够覆盖一次完整拖动路径，否则应适当提高上限。
   - 第一阶段可以先通过降低衰减和提高上限快速止血；第二阶段应改为持久化擦除 mask，不再依赖点数组长期保存拨雾结果。
4. 修改提示文案为”按住拖动拨开江雾”。
5. 完成后提示改为”江雾已开，入江寻艾”。

这一阶段能快速改善用户理解。

### 第二阶段：明确主路径

目标是让首页成为完整、可预测的入口交互。

1. 进度累计改为基于拖动距离。
   - 当前 `addFogCut` 里每添加一个点就 `+= progressStep`，拖动速度快时补点多、进度飙升，速度慢时进度推进缓慢，体验不一致。
   - 改为在 `addDragFogCuts` 中基于实际拖动 `dist` 累加 progress（例如 `progress += dist * distanceProgressFactor`），不再在 `addFogCut` 中累加。这样用户拖动相同距离获得相同进度，体验可预测。
   - 单击只给视觉反馈，不累计完成进度；或单击累计上限不超过总进度的 5%，且不能触发完成。首页主动作必须保持为“按住拖动”。
2. 新增当前动作进度线或光痕进度。
3. 完成后展示独立 CTA。
   - 当前 `dom-layer.js` 没有独立 CTA 元素，完成后仅调用 `showHint`。需要在 HTML 中新增一个 CTA 按钮（默认 `display: none`），完成后显示并绑定 `manager.goTo(1)`。
   - 样式需要 `pointer-events: auto`、`cursor: pointer`，z-index 高于 canvas 层。
   - 支持键盘 focus 和 Enter 触发。
4. CTA 点击后进入”寻艾”。
5. 底部章节节点保留为辅助导航。
6. 完成后锁定雾层打开态。
   - 建议将拖动路径写入持久擦除 mask，完成后直接渲染打开态，而不是继续依赖 `fogCuts` 点数组和生命周期。
   - 可以保留环境薄雾漂移，但主雾幕的核心开口不可回盖。
7. 已完成首页拨雾后，再次进入首页时直接展示完成态（雾层打开 + CTA 可见），不要求重复交互。
   - 当前 `onEnter` 每次重建 container 并将 `progress` 重置为 0。需要在 `SceneManager` 或 `IntroScene` 中读取已完成状态，若已完成则跳过交互直接渲染完成态。
   - DOM 状态也要同步恢复：`hero-copy` 不再 `is-muted`，hint 隐藏或切换为完成文案，CTA 显示，主拨雾 pointer 交互可禁用或只保留轻微氛围反馈。

这一阶段完成后，首页交互会从”探索隐藏规则”变成”完成一次清楚的入江仪式”。

### 第三阶段：优化仪式感

目标是在清晰易懂的基础上恢复诗意。

1. 拖动光痕可以带出水纹。
2. 雾开后远处艾草剪影出现。
3. CTA 与艾草剪影或水流方向绑定。
4. 3 秒无操作时播放一次轻微方向引导动画。
5. 引导动画只演示方向，不自动完成。

## 推荐落地版本

综合成本和体验，推荐采用以下版本：

### 交互

- 初始显示提示：`按住拖动，拨开江雾`
- 鼠标 hover 只显示弱光痕，不擦开主雾。
- 用户按住拖动后，雾沿路径散开并保留。
- 有效拖动累计到阈值后完成。
- 完成后显示：`入江寻艾 →`
- 点击后进入第二幕。

### 视觉

- 保留背景薄雾漂移。
- 主雾开口不回盖。
- 进度反馈使用细水纹，不显示数字百分比。
- 完成态让标题、副标题、下一幕线索同时清晰。

### 文案

- 初始：`按住拖动，拨开江雾`
- 拖动中：`江雾渐开`
- 完成：`江雾已开，入江寻艾`

## 验收标准

改造完成后，用第三方用户视角检查：

1. 用户 3 秒内能判断首页需要“按住拖动”。
2. 鼠标单纯划过不会被误认为已经完成任务。
3. 拖动 1 秒内能看到稳定、可累计的正反馈。
4. 松手后已拨开的核心区域不会马上被盖住。
5. 完成条件达成后，下一步入口明确可点击。
6. 不看代码、不读说明文档，也能知道如何进入下一幕。
7. 章节进度点只是辅助导航，不承担首页主 CTA。

## 不建议采用的方案

### 1. 继续加强 hover 风吹雾散

会进一步放大“划过”和“拨开”的混淆，不适合作为首页主入口。

### 2. 要求用户画出固定形状

虽然更像游戏，但会提高失败率，也不符合首页低门槛进入的目标。

### 3. 完成后自动立刻跳转

会牺牲首页完成态的观赏时间，也会让用户不确定自己是否完成了动作。本方案不采用自动跳转，统一由“入江寻艾” CTA 进入下一幕。

### 4. 把说明写成长段文字

问题不在于说明不够多，而是交互状态不够清楚。首页应该靠控件状态、反馈和文案短句解决理解问题。

## 结论

首页的核心改造方向不是增加更多效果，而是减少交互歧义：

> **把 hover 还给氛围，把拖动定义为唯一主动作，把拨开的结果稳定保留下来，并在完成后给出明确的“入江寻艾”入口。**

这样既能保留”拨开江雾”的诗意，也能让用户用最少学习成本进入后续章节。

## Review 备注

以下为代码审查时补充的实现细节，供开发阶段参考：

### 参数对照表（第一阶段）

| 参数 | 当前值 | 建议值 | 说明 |
| --- | --- | --- | --- |
| wind `centerAlpha` | 0.22 | 0.08~0.10 | 降低 hover 擦除可见度 |
| wind `midAlpha` | 0.12 | 0.04~0.05 | 配合降低 |
| `fogCutLifeDecay` | 0.00018 | 0~0.00002 | 拨开的雾不再快速回盖 |
| `maxFogCuts` | 90 | 150~200 | 第一阶段止血参数；第二阶段应改为持久 mask |

### 进度累计改造要点（第二阶段）

当前进度按”雾点数量”累加（`intro-scene.js:166`），拖动速度影响体验一致性。改为按距离累加时需注意：

- 在 `addDragFogCuts` 中计算本次拖动 `dist`，用 `progress += dist * factor` 累加。
- `addFogCut` 中移除 `this.progress += config.progressStep`，仅负责视觉擦除。
- 单击（`onPointerDown`）只提供视觉反馈，不直接推进主进度；如保留少量进度，需设置总量上限并禁止单击触发完成。
- 完成进度只由有效拖动距离主导，保证用户拖动相同距离获得相同结果。

### 持久 mask 改造要点（第二阶段）

如果只把 `fogCutLifeDecay` 降到 0，旧点仍会因为 `maxFogCuts` 上限被淘汰，长路径下仍可能出现“拨开又盖上”。第二阶段应新增持久化拨雾结果：

- 拖动时将主拨雾路径写入一个 persistent mask canvas。
- 每帧绘制雾层时先应用 persistent mask，再叠加 hover windPoints。
- `fogCuts` 可以只作为拖动中的短暂光晕或边缘柔化，不再承担长期状态。
- 完成后直接渲染完成态 mask，主雾幕核心开口固定保留。
- 再次进入首页时读取已完成状态，跳过初始雾幕，直接显示完成态。

### CTA 元素结构建议（第二阶段）

```html
<!-- index.html 中 #hero-copy 同级 -->
<button id="intro-cta" class="intro-cta is-hidden" aria-label="入江寻艾">
  入江寻艾 →
</button>
```

完成时由 `DOMLayer` 控制显隐，点击绑定 `manager.goTo(1)`。

完成态 DOM 同步规则：

- `hero-copy` 移除 `is-muted`。
- `hint` 隐藏，或切换为 `江雾已开` 后淡出。
- `intro-cta` 显示并允许 pointer / keyboard 操作。
- 底部章节进度点继续作为辅助导航，不替代 CTA。
