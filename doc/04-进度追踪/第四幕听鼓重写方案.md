# 第四幕「听鼓」重写方案

## 背景

第四幕 `04 听鼓` 当前已经具备 MVP 主流程：用户点击鼓面，生成朱砂鼓波，龙舟按点击次数向前推进，八击后过江并进入后续分幕。本轮落地时调整为 4 次击鼓完成，并改为鼓手第一视角。

本轮不直接改代码，也不替换逐幕开发文档。目标是先把第四幕剧本重写清楚，作为后续设计和实现的统一依据：它要承接第三幕「裹青」末尾的远处鼓波，把静态手作自然转入江上竞渡的动感高潮；同时完成后用龙舟水痕带出字粒，进入第五幕「问诗」。

第四幕不是竞速小游戏，也不是节日赛事海报。它应当像用户站在龙舟船头、面向前方江面亲手击鼓：敲一下，船往前行进一步；雾中舟影被唤醒，众桨随节奏合流，最后把诗字留在水痕里。

## 当前问题

1. **叙事承接偏弱**
   现有设定主要是“点击鼓面推动龙舟”，还没有充分承接第三幕完成态里的淡朱砂鼓波。用户从桌前包粽进入江面竞渡时，场景转换还需要更明确的“远鼓入江”线索。

2. **高潮层次不足**
   当前流程以点击次数为主，缺少从蓄势、起鼓、合桨到过江的分段节奏。第四幕作为全篇最快的一幕，需要有明显的动势递进，但不能变成游戏关卡。

3. **游戏化表达偏重**
   “鼓声 1 / 8”“连击”等提示容易让体验像计分任务。第四幕应保留节奏点击的手感，但弱化数字、分数、倒计时、失败和夸张连击反馈。

4. **诗句呈现不够贴合动作**
   规范里要求听鼓的诗句应出现在鼓波或龙舟水痕中，一闪而过，不宜长读。现有文档只保留短句和轻知识，没有说明诗句如何跟随鼓波出现。

5. **完成态与第五幕连接不足**
   第五幕「问诗」的入口是水中字粒。第四幕完成后应让龙舟驶过留下水痕，水痕中浮出字粒，而不是只靠底部入口进入下一幕。

## 重写目标

- 把第四幕定位为全篇从“合”到“动”的高潮转折：第三幕把愿望包入青叶，第四幕把众人节奏送回江上。
- 保留一个主动作：点击鼓面或鼓点；其他表现都由自动演出完成，不增加额外玩法。
- 让用户感到“随鼓而行”，而不是完成“点击八次”的任务。
- 视觉上延续雾青江光、暖金余晖和水墨留白，只让朱砂红成为鼓点、红绸和水痕中的短促强调。
- 完成后明确生成水痕字粒，承接第五幕「问诗」。
- 保持静态站约束：普通 `<script>` 顺序、UMD 全局对象、`window.ChuJiang` 命名空间，不引入远程资源。

## 新版剧本

### 1. 入场蓄势

第三幕的窗外淡朱砂鼓波没有立刻变成龙舟，而是沿着水纹扩散到整片楚江。画面从临水屋舍切回江面：雾青水面转入稍深的暮色，远山云气仍保留雾白和暖金余光。

画面采用从船头往前看的鼓手第一视角。下方近景出现龙舟船头、船舷透视线和一面摆在船头的传统鼓，鼓面不是舞台道具，而像用户正站在船头击鼓。前方江面是龙舟赛水路，左右岸有远景围观人群剪影和克制旗影，远处有低对比龙舟竞渡，不急着完整显露。

入场蓄势总时长控制在 600ms 至 900ms，不超过 1s，维持全篇最快一幕的节奏预期。

入场短句淡入：

> 远鼓入江，舟影渐明。

底部动作提示使用统一江面题签：

> 随鼓而行

### 2. 击鼓推进

用户点击鼓面或鼓点区域后，鼓面轻微下沉又回弹，朱砂红鼓波从鼓面向江面扩散。波纹不是大面积染红，而是一圈短促的红线，扩散后很快融入雾青水纹。

每一击都让船头轻推，江面、水痕和远处舟影向前推进一小段，同时水面出现一两笔桨影。推进不需要精确模拟比赛速度，重点是“鼓声发令，船往前行”。

鼓波中可以短暂浮现一个或两个动作字，如“起”“急”“渡”“破”，随波纹淡出。字不固定停留，不形成大字幕，也不抢第五幕的名词字粒。

第 2、3 次击鼓时，水面边缘短暂浮现两条轻知识：“鼓为令，众桨同频。”、“竞渡从江上追思，成为五月节庆。”。它们只停留约 2 秒，不阻断击鼓节奏。

### 3. 众桨合流

当用户连续点击形成稳定节奏时，表现不使用“连击”文字，而通过画面加强：

- 鼓波半径略微变大，透明度仍克制。
- 龙舟旁的桨影增多，像众人开始合力。
- 红绸在舟尾或舟首短暂一闪，作为节庆动势。
- 水痕从断续变得连贯，形成一条由左向右的推进线。

实现口径：连续 3 次点击的相邻间隔都落在 300ms 至 800ms 范围内，即视为稳定节奏，触发桨影增强和更连贯的水痕。这个阈值只用于视觉增强，不影响完成条件。

这一段的情绪是全篇最有动感的地方，但仍然不出现分数、排名、计时、失败惩罚或夸张爆闪。

### 4. 龙舟过江

达到完成条件后，最后一声鼓波比前面更开阔，但仍保持朱砂克制。龙舟从雾中完整掠过画面中景，不必占满画面，也不做正面冲撞镜头。

龙舟驶过后，江面留下长长水痕。水痕边缘有少量朱砂红退成淡金和雾白，像鼓声余韵落回江水。

完成短句淡入：

> 鼓声已渡

水痕旁出现一个小光点。用户点击光点后，轻知识从水痕边缘浮出：

> 龙舟竞渡以鼓为令，众桨同频向前。它从江上追思与乡人合力的传说中生长出来，后来成为端午最有节奏感的节庆场面。

### 5. 水痕浮字

完成态停留时，龙舟水痕中浮出 3 到 5 枚字粒，例如“江”“艾”“舟”“鼓”“风”。字粒像从水下被鼓波托起，不急着组成诗。

这些字粒是第五幕「问诗」的入口预告：用户点击「入下一幕 →」后，水痕扩散为深青静水，字粒重新排列到第五幕的可选字阵中。

## 交互与动效规则

- 主点击区域仍以鼓面为核心，但命中范围应大于视觉鼓面，保证用户不用精确点中鼓皮。
- 点击反馈控制在 120ms 至 200ms：鼓面轻压、波纹生成、龙舟短促推进同时发生。
- 鼓面点击时触发 Web Audio 短促低频包络音，约 120ms，与鼓波视觉同步；龙舟过江时可叠加一次更开阔的低频衰减。
- 完成条件默认使用 4 次点击，但界面不显示 `1 / 4` 这种任务式计数。
- 4 次只作为内部阈值，用户感知到的是“鼓声渐远、水路向前”的视觉进度，而不是数字倒计。如果后续测试发现 4 次偏多或偏少，可按体验调整，不影响界面表现。
- 稳定节奏只通过波纹、桨影和龙舟位移增强表现，不显示“连击”“combo”等游戏化文案。
- 允许用户点击水面鼓点触发，但不新增多个交互对象；水面鼓点只是鼓面的视觉延展。
- 播放过程中不设置失败状态，也不因节奏不准倒退龙舟。
- 完成后先让龙舟过江、水痕成立，再出现底部「入下一幕 →」入口。
- 轻知识沿用交互串联里的“随物解锁”模型：完成后水痕上出现小光点，用户点击光点才浮现文字；不点击不阻塞进入下一幕。
- 如果用户不点击入口，画面保持完成态：水痕、字粒、小光点和完成短句都可读。

- 红绸层：极少量朱砂红，用于龙舟动势，不铺满画面。

### 风格边界

- 江面可以比前三幕更深，但必须保留雾青、浅水青、雾白和暖金余光。
- 朱砂红只作为节奏点缀，不把整幕变成红色节庆海报。
- 龙舟保持剪影、水墨和远景感，不做 Q 版、卡通、体育赛事或游戏 UI。
- 鼓面不做舞台灯光，不出现现代音游界面。
- 字粒不做大标题，不抢过鼓波和龙舟。

## 与前后分幕的衔接

### 承接第三幕「裹青」

- 第三幕完成态的淡朱砂鼓波是第四幕入场的第一道水纹。
- 不再强调粽形强行变船头，避免形态承接过硬。
- 第四幕开场用“远鼓入江”承接桌前手作后的窗外声响，让空间从屋内自然回到江面。
- 注意：本方案与 `doc/01-项目原则/交互串联.md` 第五节“粽叶轮廓变成龙舟船形”的物象变形描述不一致。后续确认本方案后，需要同步更新交互串联文档，避免追踪时漏掉这处原则调整。

### 引出第五幕「问诗」

- 龙舟驶过后留下长水痕。
- 水痕里浮出少量字粒，但不让用户在第四幕里选字。
- 点击「入下一幕 →」后，水痕扩散为第五幕深青静水，字粒进入问诗交互。

## 验收要点

- 入场能看出承接第三幕的淡朱砂鼓波，而不是突兀切到独立竞渡画面。
- 画面仍属于雾青江光体系，不突然全黑，不变成高饱和红色节庆海报。
- 用户只需要点击鼓面或鼓点区域，不需要理解复杂规则。
- 点击反馈即时有力，但不出现分数、倒计时、失败、排名或夸张连击文案。
- 第一视角推进有节奏递进：起鼓、合桨、过江三个层次能被看见。
- 连续 3 次点击间隔都在 300ms 至 800ms 时，应能触发桨影增强；不满足时仍可正常推进，不倒退。
- 朱砂红只用于鼓波、红绸和水痕局部强调，不能污染整个画面。
- 诗句必须跟随鼓波或水痕出现，不做单独大段字幕或信息卡片。
- 轻知识通过水痕光点点击解锁，不自动淡入，不阻塞进入下一幕。
- 鼓声视觉和 Web Audio 低频包络音同步，声音短促克制，不压过画面。
- 完成态先成立，再显示「入下一幕 →」入口；不自动硬切。
- 水痕字粒能明确引出第五幕「问诗」。
- 后续实现必须保持 `file://` 本地打开可用，不依赖远程素材或模块加载。

## 落地修复经验

第四幕从方案进入实现后，暴露出的主要问题不在剧本，而在高频点击动画的状态管理。后续做第五幕、第六幕或继续打磨本幕时，可复用以下经验：

- 高频点击动画必须以“进入场景时的基准状态”为准，不要从正在 tween 的当前值继续叠加。鼓面按压、船头推进、背景视差都应记录 base 值。
- 同一对象的同一属性只能有一个明确接管者。最后一击进入完成态前，要停止点击反馈留下的 tween，再播放完成退场。
- Pixi 对象和 DOM 对象属性模型不同。Pixi 缩放写 `scale.x/y`，不要使用 `scaleX/scaleY` 这类 DOM / CSS 习惯字段。
- 循环创建的 Pixi 临时对象要在当前闭包里销毁，不能让 `onComplete` 依赖外层循环变量。
- 文案和字粒配置要防空。配置为空时直接跳过对应效果，不能让空数组取模或空文案进入 `PIXI.Text`。
- 完成态轻知识属于可选路径，`knowledgeText`、`knowledgeDot` 等对象调用前必须判断存在，避免快速切场或恢复完成态时抛错。
- Hint 文案由状态边界统一管理：进行中只显示动作提示，完成态只由完成函数设置完成短句，避免最后一帧闪烁。
